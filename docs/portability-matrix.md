# ReaShoot portability matrix

This matrix tracks what must move behind platform adapters before ReaShoot can support Windows and later Linux. The current shipping implementation remains macOS-only.

| Subsystem | Current macOS implementation | Windows target | Linux-ready decision |
| --- | --- | --- | --- |
| REAPER plugin entry/action registration | `src/reaper_video_recorder.mm` exports `ReaperPluginEntry`, loads the REAPER API, registers custom actions, `hookcommand2`, `toggleaction`, timer, and `atexit`. | Reuse the REAPER Extension SDK from a `.dll`; start with a diagnostic action before porting the full controller. | Keep action/controller code in C++ that only depends on REAPER SDK types. |
| Transport/timer/media insertion | Monolithic Objective-C++ code polls REAPER transport, tracks record start, creates the `ReaShoot` track, inserts a single `.mov`, and schedules best-effort alignment. | Move transport/media logic into platform-neutral C++ behind a small REAPER host wrapper. | Avoid OS-specific path and time handling in the controller layer. |
| UI and docking | AppKit views/controllers, Cocoa windows, native controls, and REAPER docker integration live in the Objective-C++ file. | Implement a child `HWND` preview panel with Win32 common controls and REAPER docker/floating behavior. | Keep a `PreviewPanel` interface so Linux can use SWELL, GTK, or another thin native adapter later. |
| Live preview | The plugin uses bundled `LiveKitWebRTC.framework` and `LKRTCMTLVideoView`; iPhone signaling uses the existing authenticated control WebSocket. | Prototype a desktop WebRTC receiver and render into a standalone Windows window, then embed the renderer in the REAPER panel. | Do not couple signaling to Direct3D; keep receiver/render abstractions separate so Linux can use libwebrtc, GStreamer, or another renderer. |
| Playback preview | AVFoundation `AVPlayer` / `AVPlayerLayer`, muted so REAPER remains the audio source. | Decide whether Windows MVP requires playback preview; if included, use Media Foundation or the selected preview/video stack. | Treat playback preview as `IPlaybackPreview`, separate from live preview. |
| Helper commands | Swift `video-sync-mac` helper, Foundation/Darwin sockets, `NetService`, `/usr/bin/dns-sd`, and stdout protocols consumed by the plugin. | Reimplement helper behavior in a portable core/helper; preserve current stdout initially and add JSON output for future direct/IPC calls. | Prefer C++ core first to minimize plugin/helper friction; avoid APIs that block Linux. |
| Process execution | `NSTask`, `NSPipe`, `NSFileHandle`, dispatch queues, and macOS bundle-relative helper lookup. | Hidden `CreateProcessW` adapter with UTF-16 paths, timeout/cancel support, and concurrent stdout/stderr draining. | Later POSIX adapter should use `posix_spawn` or similar, not shell-dependent command strings. |
| Download/pending/delete | Swift helper performs HTTP range resume, checksum verification, `transferComplete`, and pending-recording operations. | Move WebSocket, HTTP range download, checksum, and command formatting into portable tested code. | Manual host/port/token paths remain first-class so mDNS is optional. |
| Discovery | Helper uses `NetService` and `/usr/bin/dns-sd` for Bonjour. | Add mDNS abstraction using Bonjour for Windows if available or a bundled cross-platform mDNS library. | Leave room for Avahi or a bundled mDNS implementation. |
| Audio alignment | REAPER peak/accessor correlation plus AVFoundation movie-audio extraction fallback. | Keep REAPER peak/accessor logic and replace AVFoundation fallback with FFmpeg/libav or another cross-platform extractor. | Expose fallback through `IMediaAudioExtractor`. |
| Settings | REAPER `GetExtState` / `SetExtState`, plus macOS defaults implied by paths/UI state. | Continue using REAPER extstate so project/workflow settings behave consistently. | Avoid registry/plist-only settings. |
| Logging | `/tmp/reashoot_debug.log` plus preview status labels. | Use `%TEMP%\reashoot_debug.log`; keep routine status in UI and real errors visible. | Route all desktop logging through a tiny portable logger with platform path adapters. |
| Build/install | Makefile builds a macOS `.dylib`, Swift helper, copies `LiveKitWebRTC.framework`, and codesigns into `~/Library/Application Support/REAPER/UserPlugins`. | Add CMake targets for Windows `.dll` skeleton and future helper/core artifacts, installing into `%APPDATA%\REAPER\UserPlugins`. | Keep CMake target layout platform-explicit and allow Linux compile-only targets early. |
| Packaging | Local macOS install only. | Produce a zip/installer with the plugin DLL, helper exe, WebRTC/runtime DLLs, docs, and smoke-test checklist. | CMake install rules should describe runtime dependencies instead of one-off scripts. |

## First migration boundary

The safest first implementation boundary is a Windows plugin skeleton that loads in REAPER and registers a diagnostic action. That proves the SDK, compiler, target name, install location, and REAPER action registration before any UI, WebRTC, helper, or media insertion code is moved.
